# 架构说明

## 1. 重构目标

这次重构不是简单换一套页面，而是把项目从早期的轻量演示骨架，推进到更贴近开题报告的结构表达：

1. 技术栈对齐到 `Java + Spring Boot + Spring Cloud Gateway + MySQL`
2. 模块边界围绕开题报告中的核心功能重新组织
3. 前端门户从“概念看板”调整为更正式的答辩与管理平台首页

## 2. 当前架构

```text
Browser
   │
   ▼
Web Portal (apps/web, Nginx)
   │
   ▼
Gateway (Spring Cloud Gateway)
   ├─ user-service
   ├─ volunteer-service
   ├─ activity-service
   └─ community-service
   │
   ▼
MySQL 8
```

## 3. 服务职责映射

### gateway

- 聚合各业务服务的模块快照
- 输出门户首页使用的统一接口 `/api/portal`
- 承担统计分析模块在首页层面的汇总职责

### user-service

- 管理管理员与志愿者账户状态
- 提供角色分布、实名审核、账号状态概览
- 对应开题报告中的用户管理模块

### activity-service

- 管理活动申请、审核结果和活动发布信息
- 提供活动分类、活动排期、近一周报名趋势
- 对应活动申请管理模块和志愿活动管理模块

### volunteer-service

- 管理报名记录、签到签退、服务工时和积分
- 管理积分商品、兑换订单和志愿者画像
- 对应活动报名与服务记录管理模块、积分商城与兑换管理模块

### community-service

- 管理公告、信息动态、论坛内容和分类统计
- 承接内容互动、公告发布和社区传播展示
- 对应内容互动与公告管理模块

## 4. 数据组织方式

为了让演示工程在不引入完整业务实体的前提下仍然具备“可看、可讲、可运行”的状态，当前采用了下面的做法：

- 单个 MySQL 实例承载 4 个业务数据库
- 每个服务只访问自己的数据库，保持边界清晰
- 通过 `infrastructure/mysql/init.sql` 初始化演示表和首页所需数据
- 首页统计图表来自真实接口返回，而不是写死在前端页面里

## 5. 首页信息结构

门户页围绕答辩时最容易被追问的几个问题展开：

1. 这个平台解决什么问题
2. 模块怎么拆
3. 业务流程如何闭环
4. 统计分析是否能支撑运营视角
5. 技术栈是否与开题报告一致

因此首页被组织为：

- 概览指标
- 业务闭环
- 功能模块矩阵
- 统计分析图表
- 运营视图
- 架构落点

## 6. 与开题报告的一致性

当前工程已经对齐的点：

- 技术栈：Spring Boot、Spring Cloud Gateway、MySQL
- 模块方向：用户、活动申请、志愿活动、报名记录、内容公告、积分兑换、统计分析
- 页面要求：简洁、正式、适合作为课程设计或答辩展示

当前仍保留演示性质的点：

- 没有接入完整登录鉴权
- 还没有实现真实审批流、评论流和收藏流
- 暂未引入注册中心、消息队列、缓存与监控
- 数据以初始化演示数据为主，尚未转为真实业务写入链路

## 7. 下一步建议

如果要继续往正式毕业设计系统推进，建议优先做下面几项：

1. 补登录、JWT 和 RBAC
2. 把活动申请、审核、发布、报名、打卡做成真正可写入的流程
3. 为积分商品、订单和签收补全增删改查
4. 为论坛、评论、公告补上审核和状态流转
5. 引入 Redis、注册配置中心和链路监控
